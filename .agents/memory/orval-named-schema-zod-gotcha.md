---
name: Orval named-schema zod codegen gotcha
description: Named OpenAPI component schemas referenced only via $ref in a response do not get a zod const generated by orval — only a TS interface.
---

When an OpenAPI response body is defined as a **named component schema** and referenced purely via `$ref` (e.g. `responses.201.content.application/json.schema.$ref: '#/components/schemas/EndSessionResponse'`), orval's `@workspace/api-zod` generator emits a TS interface for it but **no zod const** — so server code cannot `.parse()` it.

Inline (unreferenced/anonymous) response schemas do get an auto-named zod const: `${OperationId}Response` in `@workspace/api-zod`, and `${operationId}${StatusCode}` as the TS result type in `@workspace/api-client-react`.

**Why:** orval's zod plugin only walks response schemas that are inline in the operation; it does not synthesize a zod schema for named components used only as $ref targets in responses (even though it does for request bodies).

**How to apply:** if a response body needs server-side Zod validation, inline the schema object directly under the operation's response in `openapi.yaml` instead of extracting it to a named component — matches the pre-existing pattern used for other simple action-result responses in this codebase (e.g. `POST /games/new`'s result). Only extract to a named component if the schema is also reused as a request body or elsewhere that does get zod codegen.
