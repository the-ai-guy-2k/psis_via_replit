# PSIS — Pitch Sequence Intelligence System

A web tool for baseball pitching coaches to log pitch sequences per plate appearance and track Good/Bad/Delta outcomes, with a dashboard of best/worst sequences and season-to-date aggregates.

## Run & Operate

- Preview: the `artifacts/psis: web` and `artifacts/api-server: API Server` workflows run automatically; restart via the workflows skill if code changes aren't reflected.
- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from `lib/api-spec/openapi.yaml` after any contract change
- No database setup needed — this app intentionally uses flat-file JSON storage (see Architecture decisions).

## Stack

- pnpm workspaces, Node.js 24, TypeScript 5.9
- Frontend: React + Vite (`artifacts/psis`), wouter routing, react-query, react-hook-form + Zod
- API: Express 5 (`artifacts/api-server`)
- Storage: flat JSON file (`artifacts/api-server/data/psis_entries.json`) — no Postgres/Drizzle used
- Validation: Zod (`zod/v4`)
- API codegen: Orval (from OpenAPI spec) — generates typed React Query hooks
- Testing: standalone scenario-based logic tests (`pnpm run test:psis`), see "Scenario testing" below
- **Explicitly out of scope (do not build without a new explicit request)**: a Pitching Training Tool, "Stat Head" style stat explorer, AI-generated recommendations/insights, advanced charts/visualizations, a player database/roster system, a cloud database (stays flat-file JSON), or authentication/user accounts.

## Scenario testing (`pnpm run test:psis`)

- `lib/psis-game-logic` is a new composite lib package holding every pure PSIS rule (outs, base-state/run advancement, inning aggregation, gameId scoping) that used to live only inside `artifacts/api-server/src/lib/psisStore.ts`. `psisStore.ts` now re-exports these from the lib and keeps only file I/O (`getCurrentGameId`/`startNewGame`/`listEntries`/`appendEntry`). This split exists so both the live server and a standalone test script can import the exact same, unduplicated logic.
- `scripts/src/test-psis-scenarios.ts` (run via `pnpm run test:psis` at the repo root, or `pnpm --filter @workspace/scripts run test:psis`) imports `@workspace/psis-game-logic` directly and runs 10 scenario tests (outs incl. capping at 3, hit run-advancement, forced-advancement walks, players-left-on-base, good/bad EABR classification, good/total fraction, inning delta = good − bad − runs, completed-innings auto-advance, and New Game gameId-boundary reset). It builds synthetic in-memory `Entry` objects (no file I/O, no server needed) via a small `simulateAtBat` helper that mirrors only the trivial request-glue in `entries.ts` (assembling goodCount/badCount/delta) — every actual rule decision is delegated to the imported lib functions.
- Running the script regenerates `artifacts/psis/public/reports/PSIS_Test_Report.md` and `.json` (overwritten each run, PASS/PARTIAL/FAIL + per-assertion detail). Since they live under the Vite app's `public/` dir, they're servable at `${BASE_URL}reports/PSIS_Test_Report.{md,json}` with no server route needed.
- The Dashboard page has a "Validation Reports" card (bottom) with download buttons for both report files, reading from `import.meta.env.BASE_URL`.
- Re-run `pnpm run test:psis` after changing any game-logic rule in `lib/psis-game-logic` to regenerate the reports before considering the change done.

## Architecture decisions

- **Original spec asked for Python/Flask + a JSON file, but this workspace only supports pnpm/TypeScript artifacts (React+Vite / Express).** Rebuilt the same functionality on the supported stack: React+Vite frontend + Express backend, keeping the JSON-file storage intent (no Postgres) since the original spec emphasized simplest possible storage.
- **EABR progressive click flow (Outcome Wizard v2)**: the Tracker no longer shows all wizard choices at once. It's a fully click-gated flow: only an "Outcome" tile is shown initially → click reveals Defense/Offense → click reveals that category's outcome-type buttons → some types need one more follow-up click → once an EABR ("End of At-Bat Result") is reached, Notes + "Log PA Entry" appear. Nothing is ever saved before EABR. State lives in `track.tsx`'s `WizardState` (`started`, `outcomeCategory`, `outcomeType`, `outcomeDetail`); `isEabrReached` gates the submit button, `goBack`/`resetForm` pop or clear the path, and a `breadcrumb` array renders the current path (e.g. "Outcome › Defense › Ground Out › Double Play"). Defense top-level options: Strikeout, Fly Out, Ground Out. Offense top-level options: Hit, Walk. Follow-ups: Fly Out → catch location (Infield/Outfield); Ground Out → play result (Single Play/Double Play/Triple Play, `outcomeDetail`); Hit → hit type (Single/Double/Triple/Home Run, `outcomeDetail`). Strikeout and Walk are themselves the EABR with no follow-up. `detailOptionsForOutcomeType()` in `artifacts/psis/src/lib/outcome.ts` is the single source of truth for which outcome types need a follow-up click and what its options are — both the wizard's gating logic and its rendered buttons read from it, so adding a new branch only requires editing that one function.
- **"Run Scored" removed as a manual offense option**: it used to be a top-level Offense wizard choice with a manual 1-4 runs picker. It's been replaced by automatic base-state-driven run calculation (see "Base-state simulation" below) and removed from `OFFENSE_OUTCOMES` in `outcome.ts`. It's kept in the `OutcomeType` schema enum (and the server still accepts a manual `runsScored` for it) purely so pre-existing stored entries keep validating — the wizard can never produce it anymore.
- **Infield Out removed from the wizard**: per an approved tracker-logic update, `infield_out` is no longer a selectable Defense option (`DEFENSE_OUTCOMES` in `outcome.ts`). It (and legacy `infield_catch`) remain in the `OutcomeType` schema enum and server out-count mapping purely so pre-existing stored entries keep validating/rendering — new entries can never produce them.
- **Game Status Panel (compact bar) + 9-box line-score scoreboard**: the panel is a single-row horizontal bar (GAME STATUS | INNING | OUTS | CURRENT Δ | LOB) with shared dividers, no per-field cards — sourced from the same `GET /innings/current` data as before (Bases relabeled "LOB", same `BaseStateDisplay` component, no API change). Below it is a fixed 9-box line-score scoreboard (one box per inning, always exactly 9 regardless of how many innings have actually been played) styled with shared borders and no card/shadow styling: each box stacks a delta row (green if positive, red if negative), an "EABR Fraction" row showing `goodCount/badCount` (Good Units / Bad Units, per the official EABR fraction rule below), and an inning-number row; incomplete/future innings render blank delta+fraction rows but always show their number. Computed purely client-side in `track.tsx` (`computeScoreboardSlots`, replacing the old unbounded `computeCompletedInningTiles`) from the full `GET /entries` list, filtered to the current `gameId` (see "Game boundary / New Game" below) — an inning counts as complete once its at-bats' `outsAdded` sum to 3, mirroring the server's own completion rule without a new endpoint.
- **Game boundary / "New Game" reset**: a `gameId` concept (separate from `inningNumber`) lets the Tracker's live view (inning/outs/delta/bases/scoreboard/recent log) reset to empty without touching historical entries, because the Dashboard's season-to-date aggregates must never reset per-game. `getCurrentGameId()`/`startNewGame()`/`isEntryInGame()` in `psisStore.ts` persist the current game boundary in `data/psis_game_state.json`; `POST /games/new` (`routes/games.ts`) bumps it. `computeInningState`/`computeLatestInningState`/`resolveInningForNewAtBat` all filter/scope by `gameId` (legacy entries with no `gameId` are treated as game 1, so pre-existing data keeps rendering as "the current game" until the first New Game click). The Tracker's "New Game" button calls `useStartNewGame()` then resets local wizard/notes state and invalidates entries/dashboard/current-inning queries — it never deletes rows from `psis_entries.json`.
- **OutcomeType/OutcomeDetail schema note**: `ground_out`'s play result (single_play/double_play/triple_play) and `hit`'s hit type (single/double/triple/home_run) live in `outcomeDetail`, not as separate top-level `outcomeType` values — this replaced an earlier flat design where `double_play`/`triple_play`/`extra_base_hit`/`home_run` were standalone `outcomeType`s. Those old top-level enum values (plus `infield_catch`, superseded by `infield_out`) are kept permanently in the `OutcomeType`/`OutcomeDetail` enums purely so pre-existing stored entries keep validating — the new wizard never produces them. `describeOutcome()` in `outcome.ts` is generic (`outcomeType` label + optional `outcomeDetail` label) so it renders both old and new shapes without a branch per type.
- Server-side detail validation lives in `entries.ts`'s `VALID_DETAILS_BY_TYPE` map: `ground_out`/`fly_out`/`hit` require a matching `outcomeDetail`, everything else rejects one. `rawOutsForOutcome()` in `psisStore.ts` computes ground_out's outs from its `outcomeDetail` (single_play=1/double_play=2/triple_play=3) instead of a flat per-type lookup, since the same `outcomeType` now yields different out counts depending on the follow-up click.
- `resultCategory`/`goodCount`/`badCount`/`strikeoutCount`/`delta` are always server-computed at entry-creation time (never accepted from the client): `resultCategory` = "good" if `outcomeCategory === "defense"` else "bad"; `strikeoutCount` = 1 only if `outcomeType === "strikeout"`; for legacy `run_scored` entries `goodCount`/`badCount` are forced to `0`; for all other outcomes `delta` = goodCount − badCount (see "Official EABR unit + delta/fraction logic" below).
- **Official EABR unit + delta/fraction logic**: EABR units are strictly per-event — every good/bad outcome is worth exactly 1 unit regardless of play detail (a `ground_out`/`double_play` or `/triple_play` is still 1 good unit; a `double`/`triple`/`home_run` hit is still 1 bad unit; `outcomeDetail` is stored but never multiplies units). Each player left on base (LOB) at inning-completion is worth **1 additional good unit**, folded directly into the completing at-bat's own `goodCount` (`entries.ts`'s `baseGoodCount + (playersLeftOnBase ?? 0)`) rather than tracked as a separate LOB-units field — this keeps `computeInningState`'s `sum(entry.goodCount)` correct with no extra aggregation step (see memory: neutralize-and-bake-in-delta-trick). **EABR Delta = Good Units − Bad Units** (`runsScored` is still computed/stored per entry for display but is no longer subtracted — this replaced the old `Good − Bad − RunsScored` formula). **EABR Fraction = Good Units / Bad Units** (replaced the old goodCount/totalAtBats fraction); shown on the Tracker's 9-box scoreboard "EABR Fraction" row as `goodCount/badCount` and testable via `computeInningState`'s `goodCount`/`badCount`.
- **Base-state simulation (auto-calculated runs & LOB)**: there is no more manual runs-scored entry or a follow-up PATCH for players left on base — both are derived automatically from a simulated 1B/2B/3B occupancy that's tracked per-inning and carried at-bat to at-bat. `applyOutcomeToBaseState()` in `psisStore.ts` is the single entry point: `walk` uses forced-advancement rules (`applyWalk` — a runner only advances if the base behind them is occupied; a run only scores if the bases were loaded), `hit` uses `advanceRunners()` with a fixed base-advance count per `outcomeDetail` (`HIT_ADVANCE_BASES`: single=1, double=2, triple=3, home_run=4) applied uniformly to the batter and every existing runner (anyone pushed past 3rd scores). Defensive outs never move runners in this MVP model (no fielder's choice/sac/tag-up/steals — intentionally out of scope). The resulting `baseState` and `runsScored` are stored per-entry; `playersLeftOnBase` is computed the instant an at-bat's outs push the inning to 3 (count of still-occupied bases at that moment) and stored directly on that completing entry — no separate PATCH call. `Entry.runsScored`/`Entry.baseState`/`Entry.playersLeftOnBase` remain optional on the schema so pre-existing entries without them still validate.
- **Backward compatibility**: entries created before the Outcome Wizard existed only have the legacy flat `result` field (no `outcomeCategory`/`outcomeType`). The `Entry` schema keeps both `result` (legacy, optional) and `outcomeCategory`/`outcomeType`/`outcomeDetail`/`playersLeftOnBase`/`runsScored`/`baseState` (optional) so all shapes validate; always render outcome text via `describeOutcome()`, never read `entry.result` or `entry.outcomeType` directly in UI code.
- Best/worst sequence rankings are computed on read (grouped by exact pitch sequence string, e.g. `FB-SL-CH`), not stored — cheap given expected data volume for a single team/season.
- **RBI calculation**: `computeRbi()` in `lib/psis-game-logic` derives RBI from the same base-state simulation used for runs/LOB — for a `hit`, RBI = number of runners (including the batter on a home run) who score as a result of `advanceRunners()`; for a `walk`, RBI = 1 only when the bases were loaded (forced-in run), else 0; all other outcomes = 0. `Entry.rbi` is server-computed and stored per entry (optional on the schema for backward compatibility, like `runsScored`/`baseState`).
- **EABR bad-units formula (RBI-inclusive, no double-subtracting runs)**: `computeEabrUnits()` in `lib/psis-game-logic` sets an offensive at-bat's `badCount` = `1 (base bad unit) + rbi` — i.e. every run driven in adds one additional bad unit on top of the flat 1-unit-per-event rule, replacing the earlier pure "1 unit regardless of detail" rule for RBI-producing outcomes. `runsScored` remains stored for display only and is never separately subtracted from delta (delta is still simply Good − Bad, now naturally reflecting RBI damage through `badCount`). Non-offensive/no-RBI outcomes (strikeout, fly_out, ground_out, a walk with bases not loaded) are unaffected and keep `badCount`/`goodCount` = 1 unit.
- **Session end-of-session flow ("End Session")**: a session is a coach-initiated summary-and-reset boundary layered on top of `gameId`, not a replacement for it — ending a session always calls `startNewGame()` under the hood (new `gameId`), so it resets the Tracker's live view exactly like "New Game" does, plus additionally computes and persists a `Session` summary. `POST /sessions/end` (`routes/sessions.ts`) rejects with 400 unless the current game has at least 1 fully completed inning (checked via the same "outs sum to 3" rule used elsewhere); on success it calls `computeSessionSummary()` (`lib/psis-game-logic`) over the current game's entries — aggregating innings completed, total good/bad units, EABR delta, session EABR fraction, hits/walks/HRs/XBH/RBI/runs allowed, strikeouts, fly/ground outs, and total LOB, plus the full inning-by-inning and at-bat-by-at-bat breakdown — then appends it to `artifacts/api-server/data/psis_sessions.json` (new flat-file store, parallel to `psis_entries.json`) before returning `{ session, newGameId }`. `GET /sessions` lists all persisted sessions. The Tracker's "End Session" button (next to "New Game") is disabled until `scoreboardSlots.some(s => s.completed)` is true; on success it shows a dismissible Session Summary card and resets local wizard/notes state the same way "New Game" does.
- **Dashboard session review (replaces sequence/recent-entries widgets)**: the Dashboard no longer shows "Top Sequences", "Worst Sequences", or "Recent Entries" — the underlying `bestSequences`/`worstSequences`/`recentEntries` fields still exist on `GET /dashboard` (data not deleted, just unused by the UI) in case they're needed again. In their place, a "Pitching Sessions" card (`dashboard.tsx`) lists every saved `Session` (via `useListSessions()`, sorted newest-first client-side) as a clickable row showing date/time, innings completed, EABR delta, and EABR fraction; clicking one opens a `Dialog` (`SessionDetailDialog`) showing the full summary stat grid, an inning-by-inning breakdown (`session.inningSummaries`), and the full at-bat list (`session.atBats`, rendered via the existing `describeOutcome()`) — all sourced directly from the already-fully-populated `Session` object with no new endpoint needed.

## Product

- **Home** — landing page explaining PSIS.
- **Tracker** (`/track`) — log a plate appearance via the EABR progressive click flow (Outcome → Defense/Offense → specific outcome → conditional follow-up → EABR reached), plus notes. Shows a running "recent log" sidebar, a compact Game Status bar (inning number, outs, current delta, LOB/bases display) with a 9-box line-score scoreboard below it, a "New Game" button to reset the live game view, an "End Session" button (enabled once ≥1 inning has completed) that saves a session summary and resets the game view, and an inning-complete summary + "Start Next Inning" control once 3 outs are reached.
- **Dashboard** (`/dashboard`) — entry count, total good/bad, average delta, top 5 / bottom 5 pitch sequences by average delta, and a recent entries table.

## Inning tracking

- At-bats accumulate within an inning until 3 defensive outs are recorded, then the inning is "complete" and a new one starts on the next logged at-bat.
- `outsAdded` per at-bat is server-computed from `outcomeCategory`/`outcomeType`/`outcomeDetail` (never trusted from the client): strikeout/fly_out/infield_out = 1; ground_out depends on its `outcomeDetail` (single_play = 1, double_play = 2, triple_play = 3); all offense outcomes = 0. It is capped so an inning's total outs never exceeds 3 (e.g. a ground_out/triple_play logged at 1 out only adds 2, not 3). Legacy top-level `infield_catch`/`double_play`/`triple_play` values map to 1/2/3 outs respectively for old entries.
- `inningNumber` is also server-computed: `resolveInningForNewAtBat` in `psisStore.ts` looks at the latest inning's entries and auto-advances to the next inning number only once that inning already has 3 outs. There is no persisted "current inning" record — inning state is always derived from the `entries` array on read via `computeInningState`/`computeLatestInningState`, consistent with how dashboard aggregates are computed on read rather than stored.
- `GET /innings/current` (`computeLatestInningState`) is for **display only** and does NOT auto-advance — it must still show the completed summary for the just-finished inning. Auto-advancing only happens in `resolveInningForNewAtBat`, called when creating a new entry. Keep this split: merging the two into one function caused a bug where the completed-inning summary was hidden immediately after the 3rd out.
- **Inning-complete summary is single-phase, no PATCH**: reaching 3 outs immediately reveals the final summary — Final Inning Delta, Runners Left On Base, "Inning N Complete", and "Start Next Inning" — with no LOB question and no confirmation button. `playersLeftOnBase` and `runsScored` are already computed and stored on the completing entry at creation time (see "Base-state simulation" in Architecture decisions), so there is nothing left to ask the user or persist post-hoc. This replaced an earlier two-phase "End Inning" confirmation UI and the `PATCH /entries/{id}` endpoint it relied on — both are gone; there is no PATCH route on `/entries` anymore.
- The Tracker page tracks a local `acknowledgedInning` state so clicking "Start Next Inning" previews the next inning's empty state (0/3 outs, delta 0) client-side without a server round trip — the real `inningNumber` assignment still happens server-side on the next submitted entry.
- Inning fields (`inningNumber`, `outsAdded`) are optional on the `Entry` schema so pre-existing entries created before this feature remain valid.
- `playersLeftOnBase` is computed server-side the instant an at-bat's outs push the inning to 3 and stored directly on that completing entry (see "Base-state simulation" in Architecture decisions) — no separate PATCH call. `computeInningState` looks at the completing at-bat's `playersLeftOnBase` for the inning-level LOB stat.
- **Recent Log scoping**: the Tracker's "Recent Log" sidebar list shows only the active inning's at-bats (`entry.inningNumber === inning.inningNumber && (entry.gameId ?? 1) === (inning.gameId ?? 1)`), not the full history — computed client-side in `track.tsx` (`recentLogEntries`), no API change. It clears the instant the inning-complete summary appears (as soon as the 3rd out is recorded), before "Start Next Inning" is even clicked — so it never shows stale/previous-inning at-bats. The `gameId` half of the filter is what makes it also clear immediately on "New Game" even if the reset inning number happens to coincide with a past inning's number. The separate "Last Entry" alert box above it is unaffected and still always shows the single most recent submission regardless of inning boundaries.
